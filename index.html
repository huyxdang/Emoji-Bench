Can a model spot a mistake it has never seen before?

Emoji-Bench measures whether language models genuinely verify reasoning steps or just pattern-match on what errors usually look like. Every task lives inside a procedurally generated emoji algebra that does not exist in training data.

Design Choices

What the benchmark is trying to isolate.

The benchmark is built to separate genuine constraint evaluation from familiarity. That means every design choice pushes toward novelty, determinism, and explicit verification.

Novel formal systems, not familiar domains

Existing metacognition benchmarks usually sit inside arithmetic, algebra, or code, all of which are heavily represented in pretraining. Emoji-Bench instead invents a fresh symbolic world for each instance, so a model has to consult the rules in front of it.

If the algebra never existed before the prompt, detecting an error requires actual table lookup and rule verification rather than learned error aesthetics.

Emoji symbols are deliberate

The symbol pool is curated for single-token behavior across major model families, clear visual separation for humans, and zero mathematical baggage. A flamingo or a magnet does not smuggle arithmetic priors into the task.

That keeps the evaluation focused on rule-following, not background semantic associations.

Difficulty scales by rule load, not chain length

Every chain stays short and readable while the formal system gets denser: more symbols, more base operations, more derived operators, and more transformations.

This makes the scaling interpretable. The task gets harder because there is more to track, not because the prompt becomes a wall of steps.

Error injection probes different failure modes

E-RES tests whether a model can catch a wrong output symbol. E-INV tests whether it checks that a cited rule exists at all. E-CASC makes later steps locally valid after an early root mistake.

Together they distinguish surface fluency from real verification.

Construction

Four deterministic stages.

Each benchmark row is built from a seeded pipeline. That keeps the released dataset reproducible and makes it possible to reason precisely about what changed between conditions.

01

System generation

Sample 3 to 6 emoji symbols, generate one or two base operation tables, optionally add derived operations, and optionally generate transformations that distribute over the base ops.

02

Chain construction

Build a leftmost-innermost derivation chain so each reduction step is explicit and the intended evaluation order stays unambiguous.

03

Error injection

Emit a clean version plus up to three single-error variants. Cascading errors are recomputed downstream so local consistency cannot rescue a shallow checker.

04

Validation

Check closure, table completeness, derived-op consistency, and transformation distribution before anything gets released into the dataset.

Example

What a failure case looks like.

This is a medium-difficulty E-CASC instance. Step 2 is wrong, and the rest of the chain is recomputed from that wrong state, which is what makes the example useful.

Medium E-CASC
Symbols: {🪸, 🪩, 🧄, 🍄}

Derived op: x ⊗ y = (x ⊕ x) ⊕ y
Transform: inv(🪸)=🪩, inv(🪩)=🪸, inv(🧄)=🍄, inv(🍄)=🧄

Start: (inv(🧄) ⊗ inv(🍄))

Step 1: (inv(🧄) ⊗ inv(🍄)) → (🍄 ⊗ inv(🍄))   [by inv]
Step 2: (🍄 ⊗ inv(🍄)) → (🍄 ⊗ 🍄)             [by inv]  ← first error
Step 3: (🍄 ⊗ 🍄) → ((🍄 ⊕ 🍄) ⊕ 🍄)           [by definition of ⊗]
Step 4: ((🍄 ⊕ 🍄) ⊕ 🍄) → (🪩 ⊕ 🍄)           [by ⊕ table]
Step 5: (🪩 ⊕ 🍄) → 🪸                         [by ⊕ table]

Why this case matters

A shallow checker can look at steps 3 through 5 and see a coherent derivation. That is not enough. The model has to notice that step 2 applied the transform incorrectly, then treat the whole suffix as downstream of the first violation.

This is why E-CASC is a useful stress test: it punishes local plausibility heuristics and rewards verification from the start of the chain.

Protocol

Prompting, outputs, and scoring are intentionally tight.

The model is framed as auditing its own prior work, then forced into structured output. That keeps the measurement focused on detection and localization instead of prose style.

Prompt structure

Each prompt contains the rules, the worked derivation, and a task asking whether there is an error and, if so, what the first incorrect step is.

The wording “your working out” is deliberate. The benchmark is measuring a metacognitive auditing setup, not open-ended explanation quality.

Output format

All evaluations use structured output. OpenAI runs use the Responses API parse path, Anthropic runs use Messages API JSON schema output, and Mistral runs use JSON object response formatting at temperature zero.

Primary metric

The page emphasizes joint accuracy: the model must both decide correctly whether an error exists and identify the first wrong step. The false positive slice is included because some models look competitive only by hallucinating errors on clean rows.

Takeaways

What the first run suggests.

The initial evaluation already shows a meaningful separation between models that check constraints and models that mostly react to surface cues.

GPT-5.4 is the clear outlier

It leads every aggregate slice and posts a 0.0% false positive rate on clean chains. That combination strongly suggests systematic verification rather than broad suspicion.

Invented-rule errors are the real separator

E-INV exposes whether a model checks that a rule exists instead of merely validating the resulting rewrite. Several otherwise strong models break sharply here.

False positives are a major failure mode

Some non-reasoning models recover many actual error rows but still collapse overall by flagging clean derivations at very high rates. The benchmark makes that visible fast.

Difficulty hurts real verifiers more than weak ones

GPT-5.4 drops from easy to expert as the rule load grows. Several weaker models stay flat because they are already operating near a floor, which fits the pattern-matching hypothesis.
